SCIO 是什么？
1.SCIO 是 BOS 操作系统的系统调用协议，全称 Simple Communication Input/Output。
2.SCIO 的设计哲学是“极简 + 通用”。它不关心应用是用什么写的（HTML/JS/CSS），只定义一套标准的“问与答”格式。内核只负责听懂、执行、回复，剩下的交给应用自己。

它定义了 iframe 应用如何与 BOS 内核通信——就像 Linux 的 syscall，Windows 的 API。

一句话概括
SCIO 是 BOS 的“系统调用语言”，让 HTML 应用能向内核请求内存、读写文件、管理进程。

核心设计
1. 消息格式
SCIO 消息采用纯文本格式，避免 JSON 解析开销。格式为：
[解释语句引擎]::[函数()];
示例：
MEM new bbb=0x1&0x3
FS_READ bbb.txt
PM START BBB.html -BBB -0
参数之间用空格分隔，特殊符号（=、&、-）用于结构化表达。

为什么这么设计？
· 纯文本在 Scratch 里好解析（用“split”积木）
· 格式统一，扩展方便
· 人类可读，调试友好

3. 通信方式
iframe 应用 → 内核：parent.postMessage()
内核 → 应用：window.addEventListener('message')
iframe 应用与 BOS 内核运行在不同的 JavaScript 执行环境中。postMessage 是浏览器提供的唯一安全跨域通信方式。内核监听所有消息，根据 SCIO 格式解析并执行，然后将结果原路返回。
为什么不直接用变量共享？
因为 iframe 有严格的安全隔离，不能直接访问内核的变量或函数。postMessage 是浏览器允许的“唯一通道”。

5. 握手协议
发起方 接收方	消息	含义
应用   内核	 "你好"	我准备好了
内核   应用	 "收到"	连接建立
握手协议的目的是建立通信信道 + 确认双方就绪。流程如下：
1. 应用加载完成后，向内核发送 "你好"
2. 内核收到后，检查自己是否就绪（内存表是否初始化、文件系统是否挂载）
3. 如果就绪，回复 "收到"；否则回复 "忙" 或 "error"。
4. 应用收到 "收到" 后，开始发送真正的 SCIO 指令

超时处理：

如果应用发送 "你好" 后 3 秒内未收到回复，视为握手失败，可以重试或报错。

支持的系统调用：
类别	指令	功能
内存管理类

mem(delete -bbb);	释放句柄为bbb的所有内存块
mem(read bbb:0x1);	读取bbb的地址0x1的内存
mem(write 0x1=0000000000000000);	在地址0x1写入0000000000000000

mem(new ...)；

格式： mem(new [句柄]=[起始地址]&[结束地址]);

原理：

· 内核维护一个内存分配表（即你之前说的“双列表”：数据列表 + 元数据列表）

· 收到 new 指令后，内核在元数据列表中查找空闲区域

· 找到后，将这段区域标记为“已分配”，并关联到指定的句柄

· 返回分配结果（成功/失败）

示例解析：

mem(new bbb=0x1&0x3); 表示：


· 句柄名：bbb

· 分配地址范围：0x1 到 0x3（共 3 字节）

· 如果成功，之后可以通过 bbb 来读写这段内存


mem(delete ...)；

格式： mem(delete -[句柄]);

原理：

· 内核查找句柄 bbb 关联的所有内存块

· 在元数据列表中将它们标记为“空闲”

· 清空数据列表中的对应项

为什么用 - 前缀？

为了与 new 的参数格式统一，- 表示“释放”。


mem(read ...);

格式： mem(read [句柄]:[物理地址]);

原理：

· 从数据列表中读取该字节的值并返回

· 如果句柄没有包含该物理地址，返回无效


mem(write ...);

格式： mem(write [地址]=[值]);

原理：

· 直接根据地址定位到数据列表中的对应项

· 将值写入（覆盖）

· 注意：这里不通过句柄，是直接地址访问——这是访问内存的一种常见方式


FS_READ();

格式： FS_READ("[文件名]");

原理：

· 内核在 BFS 文件表中查找文件名

· 找到后，从数据区读取文件内容

· 将内容原路返回给应用

注意： 文件内容通常是 Base64 编码的，应用收到后需要自己解码（如图片、音频）。


FS.WRITE();

格式： FS_WRITE("[文件名]", [内容]);

原理：

· 检查文件是否已存在（存在则覆盖，不存在则新建）

· 检查内容大小是否超过 55KB（BFS 安全限制）

· 写入文件表和文件数据区

· 返回写入结果


FS.LIST;

格式： FS.LIST;

原理：

· 内核遍历文件表

· 返回所有文件名列表（用英文逗号分隔）


PM(START ...);

格式： PM(START [应用路径] -[进程ID] -[权限等级]);

原理：

· 内核在进程表中创建一个新条目

· 分配进程ID（你指定的 BBB）

· 设置权限等级（0 表示最高权限，可以调所有系统调用；数字越大权限越低）

· 加载应用（从文件系统读取 HTML/BM 代码）

· 启动应用（如果是 HTML，加载到 iframe

· 返回启动结果

为什么需要权限等级？

不同应用应有不同权限：



· 权限 0：系统应用（任务管理器、设置），以iframe里的真实浏览器模式运行

· 权限 1：普通应用（计算器、壁纸管理器），以iframe中的安全模式运行

· 权限 2：访客应用，只能读文件，不能写，以iframe中的严格模式运行


PM(DELETE ...);

格式： PM(DELETE -[进程ID]);

原理：

· 内核查找进程ID

· 终止该进程（卸载 iframe 或清除内存）

· 回收进程占用的所有资源（内存块、文件句柄）

· 从进程表中删除该条目


PM(LIST);

格式： PM(LIST);

原理：

· 内核遍历进程表

· 返回所有进程ID及其状态（运行中/暂停/僵尸）


PM(LIST [进程ID]);

格式： PM(LIST [进程ID]);

原理：

· 检查指定进程是否存在

· 返回存在/不存在


PM(send ...);

格式： PM(send -"[消息]" [目标进程ID]);

原理：

· SCIO 的进程间通信（IPC）机制

· 内核将消息转发给目标进程

· 目标进程通过自己的消息队列接收

注意： 这是异步的——发送方不等待回复，接收方在下次调度时处理消息。


格式： SYS(INFO);

原理：

· 内核收集系统状态：

  · 内存使用率（已分配/总64KB）
  
  · 文件系统状态（文件数/空闲空间）
  
  · BOS 版本号

错误处理

所有 SCIO 指令都可能失败。失败时，内核返回：

ERR [错误码] [可选描述]

常见错误码：

· ERR_NO_MEM：内存不足

· ERR_NO_FILE：文件不存在

· ERR_NO_PROC：进程不存在

· ERR_PERM：权限不足

· ERR_FORMAT：指令格式错误


安全性
SCIO 的安全性由三部分保障：

1. 握手协议：只有完成握手的应用才能发送指令\

3. 权限系统：不同进程有不同权限等级

4. 内核校验：所有指令在执行前都会校验参数合法性（如地址是否越界、句柄是否存在）

//BOS的安全性在V1.0时还并不高


与 BM 的关系

BM（BOS-Manipulate）是 BOS 的系统编程语言，换句话说：

· SCIO 是内核对外暴露的“接口”

· BM 是SCIO的指令形式
